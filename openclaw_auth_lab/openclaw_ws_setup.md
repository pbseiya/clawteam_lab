# 🦞 OpenClaw: Sandbox Isolation Architecture Guide

เอกสารฉบับนี้อธิบายหลักการ "สถาปัตยกรรมพื้นที่กักกัน" (Sandbox Isolation) เพื่อป้องกันไม่ให้โฟลเดอร์ผลงานโปรเจกต์ (Source Code) เลอะเทอะหรือถูกรบกวนด้วยไฟล์ระบบเบื้องหลังของ OpenClaw (เช่น `.openclaw/`, `.md` tracking files, ฯลฯ) 

---

## 🔍 ปัญหาที่พบในโครงสร้างเก่า (The Problem)

ในตอนแรก เราพยายามควบคุม Workspace โดยให้เอเจนต์มาทำงานที่โฟลเดอร์ของผลงานโดยตรงด้วยคำสั่ง `openclaw setup --workspace "app/"` 
แต่ผลลัพธ์ที่ได้คือ โฟลเดอร์ `app/` ค่อนข้าง **"รก"** มาก เพราะระบบโครงสร้าง OpenClaw มองว่าที่นั่นคือบ้านของมัน มันจึงพ่นไฟล์ระบบดังต่อไปนี้มาปะปนกับโค้ดโปรเจกต์ระดับ Production ของเรา:
- โฟลเดอร์ตั้งค่าและแคช: `.openclaw/`
- ไฟล์ระบุตัวตนและกฎข้อบังคับ: `AGENTS.md`, `SOUL.md`, `IDENTITY.md`
- ไฟล์กระดานทดและวงจรชีวิต: `PLAN.md`, `HEARTBEAT.md`, `TOOLS.md`

ความยุ่งเหยิงนี้ทำให้เวลามนุษย์จะเข้าไปดู Source Code หรือพุชงานขึ้น Git จะตรวจทานได้ยากลำบาก และอาจเผลอพุชเอาแคชของระบบ AI ติดขึ้นไป

---

## ✅ วิธีแก้ไข: Agent Sandbox Isolation (The Solution)

วิธีแก้ปัญหาอย่างเด็ดขาดคือการตั้ง **"พื้นที่สมอง (Sandbox)"** กับ **"พื้นที่ผลงาน (App)"** ให้แยกขาดจากกัน:

เราจะสร้างโฟลเดอร์ขึ้นมาสองถัง ถังนึงเอาไว้ขังเอเจนต์และไฟล์ความจำของพวกมันไว้ อีกถังเอาไว้เป็นเป้าหมายเก็บโค้ดเปล่าๆ จากนั้นเราเปลี่ยนคำสั่งใน Task ให้พวกมัน "ส่งออกผลงาน" จากฐานทัพของมัน เข้าไปกองไว้ในถังผลงาน

### 🛠️ ขั้นตอนการตั้งค่าใน `launch.sh`

```bash
# 1. ตั้งค่า Profile Name (ป้องกันการกวนกันระหว่างโปรเจกต์)
export OPENCLAW_PROFILE=auth-lab

# 2. เตรียมถังสองใบให้พร้อม
mkdir -p app               # ตะกร้าเป้าหมาย (สะสม Source Code สุทธิ 100%)
mkdir -p agent_sandbox     # ตะกร้าขังไฟล์สมอง (เป็น Workspace ของ OpenClaw)

# 3. ขังตัวตน AI ทั้งหมดไว้ใน Sandbox
openclaw setup --workspace "$SCRIPT_DIR/agent_sandbox"
openclaw agents add sys-planner --workspace "$SCRIPT_DIR/agent_sandbox" --model "bailian/glm-5"
...

# 4. สั่ง Task ให้โยนผลงานข้ามกำแพงออกมาที่ ../app/ 
T0=$(clawteam task create openclaw-team "System Planning" -d "Write specs. OUTPUT ALL CODE TO ../app/" | grep 'Task created:' | awk '{print $4}')
```

## 📂 โครงสร้างโปรเจกต์ที่ได้รับการแก้ไข (The Clean Architecture)

เมื่ออัปเดตตามแนวทางนี้ โครงสร้างไฟล์ในโฟลเดอร์ระดับบนจะออกมาเป็นระเบียบและโปร่งใส:

```text
openclaw_auth_lab/
├── launch.sh                      <-- สคริปต์สปอว์นฮีโร่
├── clean.sh                       <-- สคริปต์สำหรับล้างพื้นที่ (wipes app & sandbox)
├── openclaw_ws_setup.md
│
├── agent_sandbox/                 <-- 🧠 (ตะกร้าสมอง) ปล่อยให้ระบบ AI เละเทะในนี้ให้เต็มที่
│   ├── .openclaw/                 
│   ├── AGENTS.md, SOUL.md, PLAN.md, etc.
│
└── app/                           <-- ✨ (ตะกร้าผลงาน) บริสุทธิ์ 100% มีแค่โค้ดที่รันจริง 
    ├── Dockerfile
    ├── docker-compose.yml
    ├── nginx/
    └── api.py 
```

## 🔒 บทสรุปและเหตุผลเชิงเทคนิค

1. **ไม่เผชิญปัญหา Sandbox Jail:** ถึงแม้ตัวโครงเรื่องบอกว่าเรา "ขัง" เอเจนต์ไว้ แต่ตัวประมวลผล OpenClaw ในปัจจุบันไม่ได้ล็อคการขยับตัวด้วย Filesystem Path ที่รัดกุมมากนัก ดังนั้นการที่ตัวบ้านของเอเจนต์อยู่ที่ `agent_sandbox/` แต่มันถูกสั่งให้ไปจัดการระบบในเส้นทาง `../app` จึงสามารถเดินทะลุโฟลเดอร์ออกมาวางงานได้สำเร็จอย่างราบรื่น
2. **ขจัดความจำเป็นของ Workspace Symlinks:** ในอดีตเราพยายามสร้าง `setup_workspace_meta.sh` เพื่อเอาไว้ผลักซ่อนไฟล์ `.md` ผ่านการทำเนื้อหาตัวแทนจำลอง (Symlinks) ซึ่งวุ่นวาย โครงสร้างใหม่ที่คลีนตั้งแต่รากฐานนี้ ช่วยกลบปัญหาดังกล่าวไปได้ 100% ครับ
